# 人社档案智能整理系统 — 本地私有化部署硬件配置方案

> 版本：v1.0 &nbsp;|&nbsp; 日期：2026-04

---

## 一、系统概述

本系统面向人社领域的纸质/电子档案场景，提供 **智能 OCR 识别 → 自动字段提取 → AI 质量校验 → 知识问答与历史总结** 的全链路档案数字化能力。系统采用全栈私有化部署架构，所有模型与数据均运行在甲方本地网络内，**无需连接外部互联网**，满足政务信息安全与数据主权要求。

### 核心能力

| 能力模块 | 说明 | 依赖模型 |
|---------|------|---------|
| **综合识别（VL）** | 基于视觉语言模型的高精度文档识别，支持复杂版面、印章、表格 | PaddleOCR-VL-1.5 |
| **版式识别** | 版面分析 + 表格结构化 + 区域分类 | PP-StructureV3（PP-DocLayoutV3 + SLANeXt_Plus） |
| **基础文字识别** | 快速全文 OCR，适用于纯文字材料 | PP-OCRv5 |
| **智能字段提取** | 自动提取档号、文号、题名、责任者、日期、页数、密级等归档字段 | 本地大语言模型（LLM） |
| **AI 字段校验** | 规则提取与 AI 提取交叉比对，标记冲突项，辅助人工复核 | 本地 LLM |
| **批次智能整合** | 同一文档多页/多文件自动聚合，生成分组建议 | 本地 LLM |
| **知识问答** | 对历史批次档案进行证据可追溯的知识问答 | 本地 LLM |
| **历史文档总结** | 对已归档批次数据进行 AI 总结与质量报告生成 | 本地 LLM |

### 本地 LLM 部署说明

当前系统的 AI 能力（字段提取、校验、问答、总结）通过 OpenAI 兼容 API 接口调用大语言模型。生产环境中将 MiniMax 云端 API 替换为 **本地部署的开源大模型**，通过 [vLLM](https://github.com/vllm-project/vllm) 或 [Ollama](https://ollama.com/) 提供服务，仅需修改 `.env` 中的 `MINIMAX_BASE_URL` 指向本地 LLM 服务地址即可，**应用代码无需改动**。

推荐本地 LLM 候选（按显存需求排序）：

| 模型 | 参数量 | 最低显存（FP16） | 推荐量化 | 中文能力 | 适合场景 |
|------|--------|-----------------|---------|---------|---------|
| Qwen2.5-7B-Instruct | 7B | 16 GB | AWQ/GPTQ 4-bit → 6 GB | ★★★★★ | 基础字段提取、校验 |
| ChatGLM4-9B | 9B | 18 GB | INT4 → 8 GB | ★★★★★ | 字段提取、问答 |
| Qwen2.5-14B-Instruct | 14B | 30 GB | AWQ 4-bit → 10 GB | ★★★★★ | 高精度字段提取、总结 |
| DeepSeek-V2-Lite | 16B (MoE) | 20 GB | — | ★★★★☆ | 高性价比长文本处理 |
| Qwen2.5-72B-Instruct | 72B | 144 GB | AWQ 4-bit → 42 GB | ★★★★★ | 旗舰级全功能 |

## 三、硬件配置方案

### 方案一：基础版（满足核心 OCR + 基础 AI）

> **定位**：日常档案识别为主，AI 字段提取与校验可用但响应较慢，适合 **年处理量 ≤ 5 万页** 的中小规模场景。

| 组件 | 配置 | 说明 |
|------|------|------|
| **CPU** | Intel Xeon Silver 4314 (16C/32T) 或同档 | 后端服务、PDF 处理、数据库 |
| **内存** | 64 GB DDR4 ECC | OCR 模型加载 + 数据库 + 缓存 |
| **GPU** | NVIDIA RTX 4090 (24 GB) × 1 | OCR 模型 + 7B LLM（量化）共用 |
| **系统盘** | 512 GB NVMe SSD | 操作系统 + 应用 + 模型缓存 |
| **数据盘** | 2 TB SATA SSD | 档案文件 + 数据库 + 日志 |
| **网络** | 千兆以太网 | 内网访问 |
| **操作系统** | Ubuntu 22.04 LTS / CentOS 8 | |

**可达效果**：
- ✅ 三种 OCR 模式全部可用（VL / 版式 / 基础）
- ✅ 单页 OCR 识别 1~5 秒（VL 模式），基础模式 < 1 秒
- ✅ AI 字段提取可用（Qwen2.5-7B 量化），单任务 3~8 秒
- ✅ AI 字段校验（规则 vs AI 比对）
- ⚠️ 知识问答响应较慢（10~20 秒），长文档总结能力有限
- ⚠️ OCR 与 LLM 共享 GPU，高并发时需排队
- ❌ 不建议超过 3 人同时使用

**预算参考**：约 **3~4 万元**（不含显示器、UPS 等外设）

---

### 方案二：标准版（推荐 — OCR + AI 全功能流畅运行）

> **定位**：OCR 与 AI 能力均流畅运行，支持并发处理，适合 **年处理量 5~30 万页** 的标准业务场景。

| 组件 | 配置 | 说明 |
|------|------|------|
| **CPU** | Intel Xeon Gold 5418Y (24C/48T) 或同档 | 高并发后端服务 |
| **内存** | 128 GB DDR5 ECC | 充裕的模型加载与并发处理空间 |
| **GPU 1** | NVIDIA RTX 4090 (24 GB) | **专用**：OCR 模型推理 |
| **GPU 2** | NVIDIA RTX 4090 (24 GB) | **专用**：本地 LLM 推理（14B 量化） |
| **系统盘** | 1 TB NVMe SSD (Gen4) | 操作系统 + 应用 + 模型文件 |
| **数据盘** | 4 TB NVMe SSD | 档案文件存储 + 数据库 |
| **备份盘** | 8 TB HDD（RAID 1） | 定期数据备份 |
| **网络** | 千兆/万兆以太网 | 多用户并发访问 |
| **操作系统** | Ubuntu 22.04 LTS | |

**可达效果**：
- ✅ 三种 OCR 模式全部可用，VL 模式单页 1~3 秒
- ✅ OCR 与 LLM 互不干扰，可同时处理
- ✅ AI 字段提取精度高（Qwen2.5-14B），单任务 2~5 秒
- ✅ AI 字段校验实时可用，冲突检测与证据追溯
- ✅ 批次智能整合（多文件同文档聚合）流畅
- ✅ 知识问答 3~8 秒响应，支持证据引用
- ✅ 历史文档总结与质量报告生成
- ✅ 支持 5~10 人同时在线使用
- ✅ 批量处理 500 页以内文件夹可在 10 分钟内完成

**预算参考**：约 **6~9 万元**

---

### 方案三：旗舰版（大规模档案数字化 + 极致 AI 能力）

> **定位**：面向大型档案数字化项目，需要高吞吐量、高精度 AI 和多人并发，适合 **年处理量 30 万页以上** 或有复杂分析需求的场景。

| 组件 | 配置 | 说明 |
|------|------|------|
| **CPU** | Intel Xeon Gold 6430 (32C/64T) × 2 | 双路处理器，极致并发 |
| **内存** | 256 GB DDR5 ECC | 大模型 + 高并发 + 数据库缓存 |
| **GPU 1** | NVIDIA A100 (80 GB) 或 H20 (96 GB) | **专用**：本地 LLM（72B 完整精度） |
| **GPU 2** | NVIDIA RTX 4090 (24 GB) | **专用**：OCR 模型推理 |
| **GPU 3**（可选） | NVIDIA RTX 4090 (24 GB) | OCR 扩展 / 多 worker 并行 |
| **系统盘** | 2 TB NVMe SSD (Gen4) | 操作系统 + 应用 + 全部模型文件 |
| **数据盘** | 8 TB NVMe SSD (RAID 5) | 高速档案存储 |
| **备份存储** | 20 TB+ HDD (RAID 6) | 长期归档备份 |
| **网络** | 万兆以太网 | 大文件高速传输 |
| **UPS** | 在线式 3KVA | 断电保护 |
| **操作系统** | Ubuntu 22.04 LTS | |

**可达效果**：
- ✅ 所有 OCR 模式极速响应，VL 模式单页 < 1.5 秒
- ✅ 72B 大模型完整精度运行，字段提取准确率最高
- ✅ AI 字段提取 1~3 秒，校验实时，冲突自动标注
- ✅ 知识问答 2~5 秒响应，多轮追问，深度推理
- ✅ 长文档（100+ 页）AI 总结与质量诊断报告
- ✅ 批次整合支持上千份文件的复杂归档项目
- ✅ 支持 20+ 人同时在线使用
- ✅ 批量处理 2000 页以内可在 15 分钟内完成
- ✅ 可扩展为多节点集群部署

**预算参考**：约 **20~35 万元**（视 GPU 型号浮动）

## 五、软件环境要求

所有方案均需以下软件环境，通过 Docker 容器化部署，开箱即用：

| 软件 | 版本要求 | 用途 |
|------|---------|------|
| Ubuntu Server | 22.04 LTS | 操作系统 |
| Docker Engine | ≥ 24.0 | 容器运行时 |
| Docker Compose | ≥ 2.20 | 服务编排 |
| NVIDIA Driver | ≥ 535 | GPU 驱动 |
| NVIDIA Container Toolkit | 最新 | Docker GPU 支持 |
| CUDA | 12.3+（Docker 镜像内置） | GPU 计算框架 |
| PaddlePaddle | 3.0+（Docker 镜像内置） | OCR 推理框架 |
| vLLM / Ollama | 最新稳定版 | 本地 LLM 推理引擎 |
| PostgreSQL | 15+ | 业务数据库 |
| Redis | 7+ | 缓存服务 |

### 部署步骤概要

```bash
# 1. 安装 NVIDIA 驱动 + Container Toolkit
# 2. 安装 Docker + Docker Compose
# 3. 部署本地 LLM 服务（以 Ollama 为例）
ollama pull qwen2.5:14b

# 4. 配置 .env 文件
#    - 将 MINIMAX_BASE_URL 指向本地 LLM 地址（如 http://localhost:11434/v1）
#    - 设置 MINIMAX_MODEL 为对应模型名（如 qwen2.5:14b）
#    - 启用 AUTH_ENABLED=true 并设置管理员密码

# 5. 启动系统
docker compose up -d

# 6. 首次启动会自动下载 OCR 模型（约 2~5 GB），后续启动秒级完成
```

## 六、网络与安全要求

| 要求 | 说明 |
|------|------|
| **内网部署** | 系统全部组件运行在甲方内网，无需出网 |
| **首次部署** | 需临时联网下载 Docker 镜像与模型文件（约 15~30 GB），或通过离线包导入 |
| **数据隔离** | 所有档案数据、OCR 结果、AI 交互记录均存储在本地 PostgreSQL |
| **访问控制** | 内置账号体系，支持管理员审核注册、角色权限管理 |
| **传输安全** | 建议在 Nginx 前端配置 HTTPS（甲方自签证书即可） |
| **备份策略** | 建议每日自动备份 PostgreSQL 数据库 + 档案文件目录 |

## 七、扩展演进路径

1. **多节点扩展**：当处理量超过单机极限时，可将 OCR 推理、LLM 推理、Web 服务分别部署到独立节点，通过 API 网关串联。
2. **模型升级**：随着开源大模型不断进步（如 Qwen3、GLM-5 等），只需更换 LLM 容器即可获得更强 AI 能力，应用层无需改动。
3. **多语种扩展**：PaddleOCR 支持 80+ 语种，可通过配置 `OCR_LANG` 快速切换。
4. **OCR 模型微调**：针对特定档案版式（如人社专用表格），可使用 PaddleX 对版面分析模型进行增量训练，进一步提升识别精度。
